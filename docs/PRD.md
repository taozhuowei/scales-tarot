# 项目需求文档（PRD）

## 1. 项目概述

**项目名称**：AI Tarot Yes or No  
**当前阶段**：MVP 极简版

本项目提供一个纯粹、快速、带有仪式感的塔罗 Yes / No 占卜体验。当前版本聚焦单一核心流程，不包含聊天、用户体系、主题切换、设置或图鉴等非核心功能。

## 2. 核心目标

1. 用最短路径完成一次占卜。
2. 用本地塔罗数据和本地规则生成 Yes / No / 未定结果。
3. 用克制但沉浸的动画让结果“显现”，而不是让用户感到页面跳转。

## 3. 用户流程

1. 用户打开应用，直接进入首页。
2. 首页展示标题与中心占卜圆环。
3. 用户点击圆环，进入占卜流程。
4. 占卜流程依次执行：
   - 洗牌
   - 切牌
   - 抽牌
   - 翻牌揭示
5. 抽牌完成后，无需额外点击，在同一页面自动进入结果揭示状态。
6. 用户先在首屏看到：
   - “塔罗牌根据您的问题呈现出积极/消极/尚不明朗的指示”式结果文案
   - 三张牌阵
7. 用户继续下滑查看每张牌的详细解读。
8. 用户点击“再占一次”，同页重置并回到首页初始状态。

## 4. 页面与交互结构

### 4.1 单页流程结构

整个占卜体验都在 `pages/index/index.vue` 同一页面内完成，通过状态与动画切换控制不同阶段：

- `idle`：首页待开始状态
- `shuffling`：洗牌中
- `cutting`：切牌中
- `drawing`：抽牌中
- `revealing`：翻牌与揭示中
- `result`：结果展示状态

### 4.2 首页状态

- 标题：`AI Tarot / Yes or No`
- 中心视觉：神秘圆环按钮
- 装饰：羊皮纸背景、黄铜角饰、氛围光斑

### 4.3 动画状态

- 使用 `DivinationOverlay.vue` 作为全屏仪式化动画层
- 顶部三颗四角星指示当前进度
- 抽牌完成后自动淡出，不发生路由切换

### 4.4 结果状态

- 使用结果内容面板在同页内展开
- 首屏优先展示：
  - 单一结果陈述（避免重复文案）
  - 三张牌阵
- 详细牌义在下方滚动区域展示
- 支持“再占一次”同页重置

## 5. 视觉与动效约束

### 5.1 视觉风格

- 保持现有复古羊皮纸、黄铜金色、低饱和暖色调
- 不新增突兀的新配色
- 结果布局优化以信息层级和对齐为主，不改变既有美术方向

### 5.2 动效原则

- 保留现有 GSAP 动画语言与节奏
- 不改变洗牌、切牌、抽牌的整体风格
- 重点优化几何对齐、锚点统一和不同屏宽下的落位准确性（尤其是宽屏抽牌目标 Y 轴）
- 抽牌结束后自动进入结果揭示，无额外点击

## 6. 解读系统规则

### 6.1 数据来源

- 78 张塔罗牌数据来自本地 JSON
- 每张牌包含：
  - 基本信息
  - 正位含义与情感极性
  - 逆位含义与情感极性

### 6.2 判定规则

- `positive` 计 `+1`
- `negative` 计 `-1`
- `neutral` 计 `0`

### 6.3 结果生成

- 总分大于 0：`yes`
- 总分小于 0：`no`
- 总分等于 0：`uncertain`

## 7. 技术方案

- 框架：Vue 3 + uni-app + TypeScript
- 状态管理：Pinia
- 动效：GSAP
- 构建工具：Vite
- 数据：本地 JSON + 本地解读逻辑

## 9. 测试与验收

### 9.1 自动化验证

- `npm run type-check` (Vue-tsc 类型检查)
- `npm run test:unit` (Vitest 单元测试运行)

（注：本 MVP 版本删去了繁琐的集成及 E2E 测试环境，以提升敏捷开发速度与维护性）

### 9.2 单元测试覆盖

- 解读逻辑：
  - 正向结果
  - 平衡结果
- 结果页文案逻辑：
  - 结果陈述文本映射（积极 / 消极 / 尚不明朗）
  - 首屏摘要提炼
- 状态逻辑：
  - 开始占卜
  - 抽牌生成结果
  - 进入结果态
  - 重置回初始态

### 9.3 验收路径

- 手动执行验证即可保证 H5 下呈现无误

### 9.4 手动验收重点

- 抽牌结束后无需额外点击，结果自动出现
- 不发生路由跳转
- 宽屏与窄屏下三张牌的目标位置都正确对齐
- 结果页内容可以正常滚动
- “再占一次”回到同页初始状态
