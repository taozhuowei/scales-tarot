# Scales Tarot 产品需求文档

## 0. 文档索引

本文档是 Scales Tarot 项目的产品需求文档（PRD），覆盖项目目标、目标用户、产品范围、用户流程、视觉层设计、视图过渡动画、功能需求、体验要求、非功能要求与验收重点。

塔罗领域的专有术语（卡牌名、正逆位含义、四元素象征、宫廷牌等）不在本文档展开，统一维护于：

本文档对该词表为单向引用，词表不反向引用本文档术语。

## 1. 项目概述

Scales Tarot 是一个以 H5 为主要交付形态的单页塔罗体验产品。用户进入首页后，可以直接开始一次完整的占卜流程，在同一页面内完成牌阵选择、仪式化抽牌、结果揭示和文字解读。

这个产品的重点不是复杂功能堆叠，而是用尽量短的路径、稳定的反馈和一致的视觉氛围，提供一次完整、可信、可重复的占卜体验。

## 2. 术语

本章定义全文档使用的核心术语。塔罗领域的专有术语见第 2.10 节，并由 [docs/tarot_glossary.md](docs/tarot_glossary.md) 完整维护。

### 2.1 视图层抽象（abstraction layers）

1. 视图（view）：逻辑页面，由状态驱动切换，承载一种完整交互场景。
2. 容器（container）：视图内部的位置盒子，负责定位与容纳，不关心内容细节。
3. 内容（content）：容器内部的具体物，分为文字与动画两类。

### 2.2 路由（route）

1. 主路由（main route）：承载主业务流程，内部按状态切换待机视图 / 占卜视图 / 解读分栏视图 / 解读抽屉视图。
2. 兜底路由（fallback route）：应用启动时检测到网络或资源不可用时进入，仅显示兜底视图，与主路由互斥。

### 2.3 视图（view）

1. 待机视图（idle view）：主路由初始状态，用摊牌动画引导用户触发占卜。
2. 占卜视图（divination view）：触发占卜后呈现，舞台依次播放洗牌 / 切牌 / 抽牌 / 翻牌动画。
3. 解读分栏视图（reading split view）：宽屏专用，从屏幕右外侧滑入到右半屏，与占卜视图左右分栏。
4. 解读抽屉视图（reading drawer view）：窄屏专用，从底部向上弹出，高度上限不超过结果卡牌底部，可手动拖动调整。
5. 兜底视图（fallback view）：兜底路由内的唯一视图，用神秘文案与几何动画提示无法连接。

> 解读分栏视图与解读抽屉视图统称为"解读视图"，二者依据视口断点二选一呈现。后文提及"解读视图"时，按当前屏宽自动指代其中一个。

### 2.4 容器（container）

1. 标题区（title area）：承载主标题、副标题、引导文字；兜底视图的标题区写神秘提示文案。
2. 流程进度区（progress area）：占卜视图顶部，展示流程阶段进度图标。
3. 舞台（stage）：专门用来播放动画的容器，待机视图、占卜视图、兜底视图各自持有一个舞台实例。
4. 解读面板（reading panel）：解读分栏视图与解读抽屉视图共享的内容父容器，尺寸自适应；内部包含结论容器、牌义容器、解读容器三个子容器。
5. 结论容器（conclusion container）：解读面板内，展示结果倾向结论。
6. 牌义容器（card meaning container）：解读面板内，展示卡牌信息——名称与正逆位标识，不含卡面图，卡面已由结果卡牌承载。
7. 解读容器（reading text container）：解读面板内，展示逐张牌义文字，打字机动画在此播放。
8. 操作区（action area）：仅在决策阶段出现，承载再占一次 / 重试 / 回到首页等操作按钮。解读阶段（打字机动画播放中）、占卜阶段、翻牌相位、待机阶段、兜底视图均无操作区。
9. 通知（notification）：跨视图的全局浮层容器，应用运行时所有异常通过它在顶部显示，明显但不突兀，不阻塞操作。

### 2.5 内容 — 动画（animation）

1. 卡牌动画（card animation）：作用主体是卡牌或牌堆，包含摊牌动画、洗牌动画、切牌动画、抽牌动画、翻牌动画。各动画分帧详见第 7 章。
2. 几何动画（geometry animation）：作用主体是几何图形，当前包含兜底动画——以恒星与四颗 3D 立体行星组成的循环天体动画。详见第 7 章。
3. 文字动画（text animation）：作用主体是文字，当前包含打字机动画——文字在解读容器内逐字浮现。详见第 7 章。
4. 视图过渡动画（view transition animation）：视图切换时的过渡动画，详见第 8 章。

### 2.6 流程阶段（process stage）

阶段是状态机，动画是阶段内可插拔的内容，二者解耦。本项目流程分两层：应用级流程决定视图切换，占卜内动画相位决定占卜视图内进度图标的高亮。

#### 2.6.1 应用级流程（4 阶段）

1. 待机阶段（idle）：用户尚未触发占卜，待机视图展示。
2. 占卜阶段（divination）：用户已触发，占卜视图依次播放洗牌、切牌、抽牌、翻牌等仪式动画。
3. 解读阶段（reading）：占卜视图保持卡牌定格，解读视图弹出，打字机动画播放解读文本，操作区不出现。
4. 决策阶段（decision）：解读完成（打字机动画结束）后进入，操作区淡入，等用户选择再占一次或回到首页。异常情况下决策阶段操作区可能呈现重试按钮。

#### 2.6.2 占卜内动画相位（4 个）

仅在占卜阶段内部生效，对应占卜视图流程进度区的 4 个图标：

1. 洗牌相位（shuffling）：占卜舞台播放洗牌动画。
2. 切牌相位（cutting）：占卜舞台播放切牌动画。
3. 抽牌相位（drawing）：占卜舞台播放抽牌动画。
4. 翻牌相位（revealing）：占卜舞台播放翻牌动画，结果卡牌定格。

#### 2.6.3 决策阶段的出口

- 再占一次（restart）→ 回到占卜阶段（重新经历仪式动画）
- 回到首页（back home）→ 回到待机阶段
- 重试（retry）→ 异常态下重发占卜请求，仍处于解读阶段

### 2.7 UI 操作（UI action）

1. 触发占卜（trigger divination）：点击待机视图舞台进入占卜。
2. 抽屉拖动（drawer drag）：拖动解读抽屉调整高度，不越过结果卡牌底部。
3. 再占一次（restart）：解读完成后重新开始一次完整占卜，重新经历所有动画。
4. 重试（retry）：占卜请求失败时重发请求，不重新经历仪式动画。
5. 回到首页（back home）：清空状态返回待机视图。

### 2.8 主题（theme）

1. 卡牌设计（card design）：卡牌正背面图案视觉规范。
2. 颜色规范（color spec）：项目色板与明暗、对比规则。
3. 图标设计（icon design）：流程进度图标等的视觉规范。
4. 按钮设计（button design）：按钮形态、状态、动效规范。

### 2.9 数据术语（data term）

1. 卡牌（card）：单张塔罗牌。
2. 牌堆（deck）：未发出的多张卡牌的集合。
3. 分牌堆（pile）：切牌过程中分出的小堆。
4. 结果卡牌（result card）：本次占卜最终抽到的卡牌；占卜完成后留在占卜舞台展示，解读抽屉的高度上限以此为锚。
5. 牌阵类型（spread kind）：本期仅支持单张（single card）。
6. 物理视口（viewport）：屏幕物理像素信息——宽、高、安全区、状态栏。
7. UI 预算（reservations）：各区域的像素预留——标题区、操作区、卡牌间距等。
8. 布局求解（solve layout）：纯函数布局求解器，从视口与预算算出所有容器和卡牌的几何位置。
9. 抽屉几何（drawer geometry）：解读抽屉尺寸——初始高度、最大高度、宽度、对齐方向。
10. 解读区域（reading zone）：解读面板在屏幕上的物理空间。

### 2.10 塔罗领域术语

主项目术语见 2.1–2.9。塔罗领域术语（卡牌名、正位/逆位含义、四元素象征、宫廷牌等）单独维护于 [docs/tarot_glossary.md](docs/tarot_glossary.md)，由本文档单向引用。

## 3. 产品目标

### 3.1 核心目标

1. 让用户用最短路径完成一次完整占卜。
2. 让抽牌过程保留仪式感，而不是简单表单提交。
3. 让结果内容清晰、可信、易于阅读和重新体验。

### 3.2 当前阶段目标

1. 稳定交付 H5 版本。
2. 保证核心流程可用：首页进入、抽牌、出结果、重试、再占一次。
3. 建立清晰文档和协作规范，保证后续可以持续演进。

---

## 4. 目标用户

### 4.1 核心用户

- 希望快速完成一次情绪投射、关系判断或短期决策参考的用户
- 喜欢仪式感体验，但不希望被复杂步骤打断的用户
- 更偏向移动端访问，希望在单页内完成全部流程的用户

### 4.2 使用场景

- 用户想快速抽一张或几张牌，获得方向性提示
- 用户在移动端碎片时间里完成一次轻量占卜
- 用户在结果不满意或网络中断后，希望重新尝试而不是从头理解流程

---

## 5. 产品范围

### 5.1 当前范围

- 待机视图展示产品标题、引导文字与摊牌动画引导入口
- 当前正式交付牌阵：`single_card`。多牌阵（`three_card` / `cross_spread` 等）目前未交付，未来引入时需要前后端协议、布局求解器和 UI 入口一并扩展，不再保留旧版"架构占位"代码
- 进入占卜视图后自动完成洗牌、切牌、抽牌、翻牌和解读揭示
- 在同一主路由内通过解读分栏视图（宽屏）或解读抽屉视图（窄屏）展示结果结论、卡牌信息和逐张牌义
- 支持网络失败后的提示与重试操作
- 支持再占一次操作回到待机视图重新开始

### 5.2 当前不包含的内容

- 账号系统
- 历史记录与收藏
- 支付、订阅或会员能力
- 社交分享和社区功能
- 多主题切换
- 小程序正式发布

## 6. 用户流程

1. 用户进入主路由，停留在待机视图，看到摊牌动画循环播放。
2. 用户点击待机视图舞台，触发占卜。
3. 系统切换到占卜视图，按固定流程展示仪式动画：
   - 洗牌动画
   - 切牌动画
   - 抽牌动画
   - 翻牌动画
4. 翻牌完成，结果卡牌定格在占卜舞台中央。
5. 系统进入解读阶段，宽屏弹出解读分栏视图，窄屏弹出解读抽屉视图，展示结论、卡牌信息与逐张牌义。
6. 用户可选择：
   - 阅读完整解读内容
   - 在异常时通过重试恢复
   - 通过再占一次回到待机视图重新开始

## 7. 视觉层设计

### 7.1 三层抽象

1. 视图：逻辑页面。
2. 容器：视图内的位置盒子。
3. 内容：容器内的文字或动画。

例如：待机视图是一个逻辑页面，里面"扇形展开的牌堆"是一个动画（内容），动画播放在"待机视图的舞台"容器里——位置和动画解耦。

### 7.2 5 个视图与所属容器

1. 待机视图：标题区 + 舞台
2. 占卜视图：流程进度区 + 舞台
3. 解读分栏视图（宽屏）：解读面板 + 操作区，叠加在占卜视图右半屏
4. 解读抽屉视图（窄屏）：解读面板 + 操作区，从占卜视图底部弹出
5. 兜底视图：标题区 + 舞台

### 7.3 容器与内容对应

1. 标题区：文字（主标题 / 副标题 / 引导文字；兜底视图为单行神秘提示"宇宙信号微弱，暂无法接通"）
2. 舞台：动画（待机视图播放摊牌动画；占卜视图播放洗牌、切牌、抽牌、翻牌；兜底视图播放兜底动画）
3. 流程进度区：四个图标对应占卜内动画相位（洗牌 / 切牌 / 抽牌 / 翻牌），随占卜阶段内动画推进逐个高亮。不对应应用级流程阶段。
4. 解读面板：包含三个子容器
   - 结论容器：结果倾向结论文字
   - 牌义容器：卡牌名称 + 正逆位标识（不含卡面图）
   - 解读容器：逐张牌义文字 + 打字机动画
5. 操作区：再占一次 / 重试 / 回到首页 等按钮（仅解读阶段出现）
6. 通知：错误提示文字（跨视图全局，运行时异常时浮现）

### 7.4 视图与应用级流程阶段的对应

1. 待机阶段（idle）→ 待机视图
2. 占卜阶段（divination）→ 占卜视图，含全部洗牌、切牌、抽牌、翻牌仪式动画
3. 解读阶段（reading）→ 解读视图（解读分栏视图或解读抽屉视图），叠加在占卜视图之上，打字机动画播放，操作区不出现
4. 决策阶段（decision）→ 仍是解读视图，操作区淡入，等用户决策

占卜视图流程进度区的 4 个图标在占卜阶段内逐个高亮，对应洗牌、切牌、抽牌、翻牌四个动画相位，与应用级流程阶段不一一对应。

### 7.5 动画分帧

#### 7.5.1 卡牌动画（card animation）

作用主体是卡牌或牌堆。

##### 摊牌动画（fan animation）

待机视图舞台，循环播放。

- 帧 1（0%）：牌堆静态叠放在舞台中央，所有牌完全重叠
- 帧 2（30%）：牌张沿弧线向左右两侧扇形展开，每张带轻微旋转
- 帧 3（55%）：扇形完全展开成对称弧形，短暂停顿
- 帧 4（85%）：反向收拢，牌张回到中央
- 帧 5（100%）：完全合叠，进入下一循环

##### 洗牌动画（shuffle animation）

占卜视图舞台。

- 帧 1（0%）：牌堆完整叠放在舞台中央
- 帧 2（25%）：牌堆从中央分裂为左右两叠
- 帧 3（55%）：左右两叠交错向中央汇聚（Riffle Shuffle 效果）
- 帧 4（85%）：交错完成，重新合为单叠
- 帧 5（100%）：单叠归位，准备进入切牌

##### 切牌动画（cut animation）

占卜视图舞台。

- 帧 1（0%）：牌堆居中静态
- 帧 2（30%）：上半叠向上抬起并向侧方偏移
- 帧 3（60%）：上半叠落到主叠侧方，形成两个分牌堆
- 帧 4（85%）：分牌堆稳定
- 帧 5（100%）：完成，进入抽牌前预备态

##### 抽牌动画（draw animation）

占卜视图舞台。

- 帧 1（0%）：分牌堆稳定
- 帧 2（30%）：单张牌从分牌堆顶部抽出，向上抬升
- 帧 3（70%）：牌移动到舞台中心，尺寸渐渐放大
- 帧 4（100%）：牌定格在中心位置，背面朝上——此即结果卡牌位置

##### 翻牌动画（flip animation）

占卜视图舞台。

- 帧 1（0%）：结果卡牌背面朝上静止
- 帧 2（30%）：沿垂直轴开始旋转，背面渐倾斜
- 帧 3（50%）：旋转至 90°，牌仅可见侧边线
- 帧 4（80%）：旋转至 180°，正面完整呈现
- 帧 5（100%）：正逆位标识浮现

#### 7.5.2 几何动画（geometry animation）

作用主体是几何图形。

##### 兜底动画（fallback animation）

兜底视图舞台，循环播放，3D 立体优先。

- 静态元素：
  - 中央一颗静态发光恒星（球体）
  - 四条可见的椭圆轨道线，半径各异
  - 四颗 3D 立体几何行星：
    - 正四面体（tetrahedron）— 由原"等边三角"立体化
    - 平行六面体（parallelepiped）— 由原"平行四边形"立体化
    - 球体（sphere）— 由原"圆形"立体化
    - 八面体（octahedron）— 由原"菱形"立体化
- 运动：
  - 四颗行星沿各自椭圆轨道围绕恒星公转，半径与速度各异
  - 每颗行星以自身轴心持续自转
  - 缓慢、神秘、无限循环
- 不画任何行星之间的连线
- 实现技术：GSAP（CSS 3D transform + GSAP timeline + MotionPath 插件）

#### 7.5.3 文字动画（text animation）

作用主体是文字。

##### 打字机动画（typewriter animation）

解读容器内。

- 帧 1（0%）：文本区域为空
- 帧 2（进行中）：文字逐字浮现，单字符间隔约 30~50ms
- 帧 3（进行中）：光标闪烁于末位
- 帧 4（100%）：全文呈现完毕，光标隐去

## 8. 视图过渡动画

视图切换时的过渡动画分两段：从待机视图到占卜视图，从占卜视图到解读视图（宽屏分栏 / 窄屏抽屉两个分支）。

### 8.1 第一段：待机视图 → 占卜视图

核心思路：舞台与牌堆是公共且连续的；标题与进度是上下交替的；牌堆从摊牌末态直接衔接进洗牌首态，不重建。

#### 8.1.1 阶段 1 — 等待摊牌收拢

1. 用户点击舞台触发后不立刻过渡。
2. 等当前摊牌动画播放到帧 5（100%，牌堆完全合叠），停止循环。
3. 等待时长 0~800ms，取决于点击瞬间摊牌动画的相位。

#### 8.1.2 阶段 2 — 元素交替 + 洗牌启动（同步进行，时长约 450ms）

1. 标题容器：向上平移 + 透明度 1 → 0 淡出。
2. 流程进度区：从舞台上方原标题位置（向上偏移约 16~24px 处）下移到目标位置 + 透明度 0 → 1 淡入。
3. 舞台上的牌堆（摊牌收拢的同一套牌）开始洗牌动画首帧（左右分裂）。
4. 缓动曲线：cubic-bezier(0.16, 1, 0.3, 1)（先快后慢，等价 GSAP 的 expo.out 或 power4.out）。

#### 8.1.3 阶段 3 — 占卜视图就位

1. 标题容器完全消失。
2. 流程进度区稳定在顶部。
3. 洗牌动画继续推进。

#### 8.1.4 关键设计要点

1. 舞台位置不变 — 待机视图与占卜视图共用同一舞台组件实例。
2. 牌堆是同一组卡牌实例，不销毁不重建，视觉连续。
3. 标题与进度的交替不打断舞台上的卡牌动画。

### 8.2 第二段：占卜视图 → 解读视图

触发时机：翻牌相位结束（翻牌动画帧 5：100% 完成，正逆位标识浮现完毕）。翻牌完成后停顿约 300ms，让用户看清正逆位标识，再启动解读视图入场。

#### 8.2.1 宽屏分支 → 解读分栏视图

##### 阶段 1 — 占卜舞台让位 + 分栏入场（同步进行，时长约 450ms）

1. 占卜视图宽度由 100% 收缩到 50%（左半屏），缓动 cubic-bezier(0.16, 1, 0.3, 1)。
2. 占卜舞台和结果卡牌随容器收缩，结果卡牌最终落位于左半屏中央。
3. 解读分栏视图从屏幕右外侧滑入，占据右半屏。
4. 操作区容器结构挂载但保持透明度 0，等待决策阶段再淡入。

##### 阶段 2 — 解读面板内容同步呈现 + 打字机停顿启动

1. 结论容器、牌义容器、解读容器三个子容器同步淡入。
2. 解读容器入场结束后停顿 50~100ms，再启动打字机动画。
3. 此时进入解读阶段（reading），操作区不出现。

##### 阶段 3 — 决策阶段进入

1. 打字机动画完整播放结束后进入决策阶段（decision）。
2. 操作区淡入（350ms，缓动 cubic-bezier(0.16, 1, 0.3, 1)），承载再占一次、回到首页按钮。
3. 异常态下操作区改为重试按钮。

#### 8.2.2 窄屏分支 → 解读抽屉视图

##### 阶段 1 — 抽屉从底部弹出（时长约 350ms）

1. 解读抽屉视图从屏幕底部边缘向上滑入，缓动 cubic-bezier(0.16, 1, 0.3, 1)。
2. 初始高度：抽屉顶端贴齐结果卡牌底部——这是抽屉的高度上限锚点。
3. 占卜视图保持原位，不收缩、不位移。
4. 操作区位于屏幕底部，独立于抽屉，容器结构挂载但保持透明度 0，等待决策阶段再淡入。

##### 阶段 2 — 解读面板内容同步呈现 + 打字机停顿启动

1. 结论容器、牌义容器、解读容器三个子容器同步淡入。
2. 解读容器入场结束后停顿 50~100ms，再启动打字机动画。
3. 此时进入解读阶段（reading），操作区不出现。

##### 阶段 3 — 决策阶段进入

1. 打字机动画完整播放结束后进入决策阶段（decision）。
2. 操作区淡入（350ms，缓动 cubic-bezier(0.16, 1, 0.3, 1)），承载再占一次、回到首页按钮。
3. 异常态下操作区改为重试按钮。

## 9. 功能需求

### 9.1 待机视图

- 展示产品标题与副标题
- 提供明确的触发占卜入口（点击舞台进入占卜）
- 提供牌阵选择入口
- 在资源加载失败时，进入兜底视图，并通过通知或兜底标题区给出清晰提示

### 9.2 牌阵选择

- 当前版本仅开放单张牌阵（single_card）
- 默认状态下可直接进行单张牌阵占卜
- 多牌阵尚未实现；未来引入时需要在协议（`/divinations` 请求体的 `spreadKind` 枚举）、布局求解器（`SpreadKind` 类型与求解策略）和 UI 入口三处一起扩展

### 9.3 占卜流程

- 占卜流程应自动推进，不要求用户在中途多次点击确认
- 仪式动画应传递仪式感，但不能明显拖慢用户完成结果的速度
- 在减弱动态效果场景下，流程仍应可完成

### 9.4 解读展示

- 在解读分栏视图（宽屏）或解读抽屉视图（窄屏）内展示
- 结论容器显示结果倾向结论
- 牌义容器显示本次抽到的卡牌信息（名称 + 正逆位标识，不含卡面图）
- 解读容器显示逐张牌义内容，由打字机动画呈现
- 长文本应保持可读，不因内容长度导致解读区不可用
- 用户应能明确执行再占一次操作

### 9.5 异常与恢复

- 当读取卡牌资源失败时，待机视图必须给出重试路径，或进入兜底视图
- 当占卜请求失败时，必须通过通知给出清晰错误提示
- 重试时直接重新发起一次完整占卜（后端在单一事务内重抽 + 重判），用户不需要重新经历仪式动画，但抽到的牌会换一组——这与"抽牌 + 解读"已合并为一个服务端步骤的协议一致
- 非阻塞问题不能导致整页白屏或卡死

### 9.6 可访问性与可用性

- 关键操作必须具备清晰可见的入口
- 关键内容必须可被辅助阅读工具识别
- 移动端主要交互目标需保持足够触控面积
- 常见屏幕尺寸下不应出现明显遮挡、错位和跳变

## 10. 体验要求

### 10.1 视觉方向

- 整体风格保持复古、克制、稳定
- 页面氛围应服务于占卜体验，不追求花哨堆叠
- 正向和负向结果可以有差异化强调，但不能破坏整体一致性

### 10.2 交互原则

- 尽量减少不必要步骤
- 给用户明确反馈，不让用户猜系统是否还在工作
- 错误恢复路径必须简单、直接
- 解读视图要兼顾首屏重点信息和完整阅读体验

### 10.3 动效规范

1. 缓动语言：项目过渡统一采用 cubic-bezier(0.16, 1, 0.3, 1) 或同家族 cubic-bezier(0.32, 0.72, 0, 1)，先快后慢，末段柔和落定。
2. 时长基线：单段过渡动画 350~500ms；翻牌完成到解读入场之间留约 300ms 沉淀；打字机动画启动前停顿 50~100ms。
3. 操作区仅在决策阶段淡入，淡入时长 350ms，缓动 cubic-bezier(0.16, 1, 0.3, 1)；待机阶段、占卜阶段、翻牌相位、解读阶段（打字机播放中）、兜底视图均无操作区。
4. 兜底视图标题文案"宇宙信号微弱，暂无法接通"作为唯一文字提示，不再叠加 toast 或通知。

## 11. 非功能要求

- H5 为当前唯一正式交付形态
- 核心流程需要在常见移动端尺寸下稳定可用
- 核心文档必须足够清晰，使后续协作者不依赖特定 AI 工具也能继续推进

---

## 12. 验收重点

1. 待机视图到解读视图的核心链路可稳定完成。
2. 单张牌阵的展示和解读内容匹配。
3. 网络异常和服务端错误时，用户有明确恢复路径（通知提示 + 重试操作 + 必要时进入兜底视图）。
4. 再占一次操作可以回到待机视图，可再次完整开始一次占卜。
5. 产品文档、塔罗术语词表和执行计划之间职责清晰，不互相覆盖。
